<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → `/api/kpi/latest` | Untrusted query params (`device_id`, `range`) enter server runtime |
| Server → InfluxDB | Server uses secrets to access data source; must never expose token |

## STRIDE Threat Register (focus: SSRF, token leakage, internal endpoints)

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|----------|----------|-----------|-------------|-----------------|
| T-01-01 | I (Info Disclosure) | Secret token in client bundle | mitigate (HIGH) | Keep Influx code in `lib/influx/*` guarded by `import "server-only"`; never expose `INFLUX_TOKEN` via props/JSON; tests assert response has no secret keys; avoid `NEXT_PUBLIC_*` secrets. |
| T-01-02 | T/I (SSRF + Info Disclosure) | Route accepts Influx URL/org/bucket/token from client | mitigate (HIGH) | Route handler validates only `device_id` + `range` preset; Influx connection params read only from `lib/env.ts`; never branch on client-provided host. |
| T-01-03 | T (Injection) | Flux query string injection via `device_id` | mitigate (MED) | Strictly validate `device_id` charset/length; build Flux via controlled template; add unit test for validation rejects suspicious strings. |
| T-01-04 | D (DoS) | Unbounded ranges causing heavy Influx load | mitigate (MED) | Limit `range` to a small preset set; no custom start/stop in Phase 1; return `400` on invalid range. |
</threat_model>
