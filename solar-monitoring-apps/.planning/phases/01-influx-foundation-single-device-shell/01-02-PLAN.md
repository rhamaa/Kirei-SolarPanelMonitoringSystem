<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser UI → `/api/kpi/latest` | Untrusted user inputs (device_id/range) influence server query selection |
| Server-rendered `/dashboard` → client bundle | Only non-secrets may cross (e.g., DEFAULT_DEVICE_ID); secrets must not |

## STRIDE Threat Register (focus: SSRF, token leakage, internal endpoints)

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|----------|----------|-----------|-------------|-----------------|
| T-01-05 | I (Info Disclosure) | Token leakage into client via props/state | mitigate (HIGH) | Only pass `DEFAULT_DEVICE_ID` and non-secret defaults to client; never import `@/lib/env` in client modules; keep KPI DTO free of secrets. |
| T-01-06 | I (Info Disclosure) | Error UI prints secret values | mitigate (HIGH) | Config-missing callout lists required var names only; do not render `process.env` values; treat any thrown env errors as untrusted strings. |
| T-01-07 | T (SSRF-like proxy abuse) | UI enables arbitrary backend targets | mitigate (MED) | UI provides only `device_id` and `range` inputs; no URL fields; server ignores any other query params. |
| T-01-08 | R (Repudiation) | Lack of error context for operators | accept (LOW) | Phase 1 is LAN/internal; minimal stable error copy is sufficient; Phase 3 runbook will expand troubleshooting. |
</threat_model>
